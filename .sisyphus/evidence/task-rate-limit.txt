# Rate Limiting Implementation - Verification Guide

## Implementation Summary

Added server-side IP-based rate limiting to `create-cashi-payment` Edge Function:
- **Limit**: 10 requests per minute per IP address
- **Window**: Sliding 60-second window
- **Storage**: In-memory Map (suitable for Edge Functions)
- **Response**: HTTP 429 with `Retry-After` header when limit exceeded

## Key Features

1. **IP Extraction**: Checks `X-Forwarded-For` → `X-Real-IP` → fallback to 'unknown'
2. **Sliding Window**: Removes expired timestamps, only counts requests within last 60 seconds
3. **Retry-After Header**: Tells client exactly when they can retry
4. **Logging**: Warns when rate limit is exceeded with IP and request count

## Code Changes

### Added Functions:
- `getClientIP(req)`: Extracts client IP from request headers
- `checkRateLimit(ip)`: Implements sliding window rate limiter

### Modified Handler:
- Added rate limit check before processing payment request
- Returns HTTP 429 with proper headers when limit exceeded

### Manual curl test:
```bash

for i in {1..15}; do
  curl -X POST http://localhost:54321/functions/v1/create-cashi-payment \
    -H "Content-Type: application/json" \
    -H "apikey: YOUR_ANON_KEY" \

    -w "\nStatus: %{http_code}\n\n"

```

### Expected Response (429):
```json
{
  "error": "Rate limit exceeded. Too many payment requests.",
  "retry_after_seconds": 45
}
```

Headers:
- `Retry-After: 45`
- `Content-Type: application/json`

## Security Considerations

✅ **Implemented:**
- Server-side enforcement (cannot be bypassed by client)
- IP-based limiting (prevents spam from single source)
- Proper HTTP status codes and headers
- Logging for monitoring

⚠️ **Limitations:**
- In-memory storage: Rate limit state resets on function cold start
- Shared IP (NAT/proxy): Multiple users behind same IP share the limit
- No distributed state: Each Edge Function instance has separate counters

## Production Recommendations

For production deployment, consider:
1. **Deno KV**: Persistent storage across function instances
2. **Redis**: Distributed rate limiting with TTL
3. **Cloudflare Rate Limiting**: Edge-level protection
4. **Per-user limits**: Combine IP + session token for authenticated users
